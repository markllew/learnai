Module 3: Version Control with Git & GitHub

Tracking Changes and Collaborating on Code

As your projects grow, keeping track of changes, fixing mistakes, and collaborating with others becomes difficult. Version Control Systems (VCS) solve this. Git is the world's most popular VCS, and GitHub is the most popular platform for hosting Git repositories.

1. What is Version Control?

Think of version control as an unlimited "undo" button for your entire project. It's a system that records changes to a file or set of files over time so that you can recall specific versions later. It also allows multiple people to work on the same project without overwriting each other's work.

2. The Core Git Workflow

The Git workflow primarily involves three "areas" on your local machine:

  1. Working Directory: The actual files and folders you are currently working on.
  2. Staging Area: A "holding area" where you place the changes you want to include in your next "snapshot" (commit).
  3. Repository (`.git` directory): The database where Git permanently stores all the snapshots (commits) of your project.

The basic workflow is: Modify files → Stage changes → Commit snapshot.

# 1. Initialize a new Git repository in your project folder
git init

# 2. Check the status of your files
git status

# 3. Add a file to the staging area
git add my_script.py

# 4. Commit the staged changes to your repository with a message
git commit -m "Initial commit with a basic script"

# 5. View the history of your commits
git log

3. Branching and Merging

This is Git's killer feature. A branch is a movable pointer to one of your commits. The default branch is called `main` (or `master`).

You can create a new branch to work on a feature (e.g., `feature-add-auth`), and once it's complete and tested, you can **merge** it back into your `main` branch.

4. GitHub: Remote Repositories

While Git is the tool on your computer, GitHub is a service that hosts your repositories on the web. This allows you to:

  • Create a backup of your code.
  • Collaborate with others.
  • Showcase your projects in a portfolio.

The main commands are `git push` (to send your commits to GitHub) and `git pull` (to fetch changes from GitHub).

AI-Proof Exercise: Branching Workflow

  1. Initialize a Git repository for one of your previous exercise projects. Make an initial commit on the `main` branch.
  2. Create a new branch called `refactor`.
  3. On the `refactor` branch, make a significant change to your code (e.g., improve a function, add comments). Make a new commit for this change.
  4. Switch back to the `main` branch. Observe that your changes are gone.
  5. Merge the `refactor` branch into `main`. Observe that your changes have now appeared in `main`.

Challenge: After merging, you realize there was a bug in your refactor. What Git command would you use to view the commit history and identify the commit hash of the merge? Conceptually, how could you undo this merge if it was causing major problems?
